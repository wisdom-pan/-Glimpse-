### 1.2 原生 OCR：DBNet 检测 + CTC 识别（`cpp/ocr/ocr_engine.cpp`）

检测后处理完整复刻 PP-OCR 的 DB 算法，全部用 MNN-CV 算子实现：

```cpp
// 1) 概率图二值化 (kDetThresh = 0.3) -> 连通域
for (size_t i = 0; i < binData.size(); ++i)
    binData[i] = prob[i] > kDetThresh ? 255 : 0;
auto contours = findContours(binImg, RETR_LIST, CHAIN_APPROX_SIMPLE);

// 2) 最小外接矩形 + box score 过滤
RotatedRect rr = minAreaRect(cnt);
// ... 在 quad 的轴对齐包围盒内求 prob 均值作为 boxScore

// 3) DB unclip：按 area*ratio/perimeter 向外扩张文本框
float dist = area * kUnclipRatio / peri;
expanded.size.width  += 2.f * dist;
expanded.size.height += 2.f * dist;

// 4) 透视裁剪矫正 + 竖排文本旋转
Matrix M = getPerspectiveTransform(src, dst);
VARP crop = warpPerspective(bgr, M, Size(cw, ch), INTER_LINEAR);
if ((float)ch >= 1.5f * cw) {                 // 竖排 -> 顺时针 90°
    crop = _Transpose(crop, {1, 0, 2});
    crop = _Reverse(crop, _Scalar<int>(1));
}

// 5) CTC greedy 解码：逐帧 argmax，跳过 blank(0) 与重复
for (int t = 0; t < T; ++t) {
    int best = argmax(logits + t*C, C);
    if (best != 0 && best != prev) text += mKeys[best];
    prev = best;
}
```

阅读顺序还原：检测框按 `上→下、左→右` 排序（Y 差 >10px 视为换行）。

### 1.3 结构化引擎：复用 MNN-LLM 通道（`StructuringEngine.kt`）

不另起 runtime，直接用工程已有的 `ChatService`/`LlmSession` JNI：

```kotlin
val s = ChatService.provide().createLlmSession(
    modelId = modelId, modelDir = configPath,
    supportOmni = false, useCustomConfig = false)
s.load()
s.updateThinking(false)      // 关闭思考链，直出 JSON
s.updateMaxNewTokens(256)    // 限长压低延迟
// 生成时 prompt 末尾追加 " /no_think" 作为 Qwen3 硬保险
```

### 1.4 场景化 Prompt + 鲁棒解析（`StructuringPrompt.kt`）

单一 JSON 输出约束，六场景自适应（chat/meeting/card/invoice/note/other），各场景指定提取重点；解析端容忍 markdown 包裹与前后缀，失败时降级保底（取原文前 15 字 + 标记低置信度字段），从不抛异常导致崩溃。

### 1.5 模型释放（`OcrModelInstaller.kt`）

MNN `Module::load` 需要真实文件路径，故首启时把 `assets/ocr/` 三个文件拷入私有目录后再加载。

## 二、技术难点与解决方案

| # | 难点 | 根因 | 解决方案 |
| :-- | :--- | :--- | :--- |
| 1 | **HEIC/Exif 图像解码不稳** | MNN 内置图像解码器对 iPhone HEIC、Exif 旋转支持有限 | 用 Android `BitmapFactory` 解码 + Exif 旋转矫正，把裸 RGB 交给 native `runRgb()`，绕开 MNN 解码器（`OcrEngine.kt`） |
| 2 | **OCR 在 Java 层做后处理太慢且依赖 OpenCV** | DB 后处理（连通域/外接矩形/unclip/透视）计算密集 | 全部下沉到 C++，复用 **MNN-CV**（`tools/cv`）算子，无需引入完整 OpenCV，控制包体 |
| 3 | **小模型输出 JSON 不稳定** | 2B 模型偶发输出推理过程、markdown 包裹、多余前缀 | `updateThinking(false)` + `/no_think` 双保险 + 正则鲁棒抽取 JSON + 解析失败优雅降级 |
| 4 | **竖排 / 旋转文本漏识别** | 中文截图常含竖排或旋转文字 | 裁剪后按宽高比判定竖排（`h≥1.5w`），`_Transpose`+`_Reverse` 旋转 90° 再送识别 |
| 5 | **多模型内存峰值** | OCR + Qwen + Gemma 多模态同驻易 OOM | 模型按需懒加载（首次用到才 `load`）、用完可 `release`；编译开 `MNN_LOW_MEMORY` + HQQ 量化 + mmap |
| 6 | **检测分辨率与精度/速度权衡** | 大图直接推理慢，小图丢字 | 长边限制 960 + 32 对齐，ImageNet 均值归一化（`toNchw`），平衡精度与延迟 |
| 7 | **隐私"可证明"而非"承诺"** | 用户需确信数据不外传 | 基础链路无任何网络调用路径；云增强为独立模块、默认关、强制脱敏、原图禁传；密钥走 Keystore |

## 三、补充说明

### 3.1 关于模型版本的诚实说明

PRD 早期文稿写作 **PP-OCRv5 / Qwen3.5-2B / Gemma 4 E2B**；工程实际采用社区可获得、可验证的真实 MNN 量化权重：

- OCR：**PP-OCRv6_small**（`det.mnn` 4.7MB + `rec.mnn` 10MB，随 APK 内置）
- 结构化：**Qwen3-2B**（HQQ 量化，经模型市场下载）
- 音频：**Gemma 3n E2B**（多模态，omni 通道）

模型选型以"端侧可跑、权重可得"为第一原则，命名以工程实物为准。

### 3.2 SME2 加速：硬件支持 ≠ 自动启用

这是部署中最易踩的坑，需三个条件同时满足：

1. **硬件支持**：`adb logcat -d | grep "device supports"` 输出含 `sme2: 1`（该日志为 CPU 硬件检测，不受编译开关影响）
2. **编译开启**：`-DMNN_SME2=ON -DMNN_KLEIDIAI=ON`，否则引擎不走 SME2 优化路径
3. **算子命中**：LLM 解码阶段的 GEMM 是主要受益算子

三者缺一，SME2 红利不生效——硬件支持只是必要非充分条件。

### 3.3 验证路径

- **命令行先验**：`llm_demo` 跑通推理、`llm_bench` 出基准，再集成进 APP（避免 APP 层问题与引擎层问题混淆）
- **`.so` 自动打包**：APP 编译自动读取 `project/android/build_64/lib/libMNN.so` 打进 APK，无需手动 push
- **真机基准为准**：性能目标（单图 ≤3s、1 分钟音频 ≤5s）需在 4GB+ RAM、Android 10+ 真机实测确认；**多模态音频在低内存机型的延迟是首要风险项**，尚待验证。

### 3.4 已知局限（不回避）

- 端侧 2B 模型对复杂长文本（如多页合同）的字段抽取精度有限——这正是 V2.0 云端增强（用户自备 Key + 强制脱敏）的适用边界。
- 30 分钟以上超长音频在端侧延迟不可控，归为云增强可选场景。
- OCR 对低质量/严重倾斜/艺术字截图仍有漏识别——通过确认页低置信度高亮交由用户校正，而非追求 100% 自动化。
