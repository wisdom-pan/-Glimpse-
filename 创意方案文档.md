# 创意方案：拾光（Glimpse）

> 基于 MNN 端侧推理引擎的隐私优先离线信息管理 Android 应用
> 「拾起碎片时光，到点自会想起」
>
> 版本 V3.1 · 2026-06-22

---

## 一、应用场景

人每天产生大量"碎片信息"：领导在微信里甩来一句"周五前把合同发我"、一张随手拍的会议白板、一段 10 分钟的客户语音留言、一张报销发票。这些信息的共同特征是：**到达时是非结构化的，价值却依赖于被结构化后及时执行**。现实中它们大多沉没在相册和聊天记录里，到点遗忘。

本方案面向三类高频场景：

| 场景 | 输入形态 | 用户真正想要的结果 |
| :--- | :--- | :--- |
| **职场指令归档** | 微信/钉钉聊天截图 | 自动析出待办 + 截止时间，写入日历提醒 |
| **会议/语音留言** | 30 分钟内录音 | 转写 + 提炼纪要、行动项、负责人 |
| **票据/名片/便签** | 拍照 | 析出金额、商户、联系人，建台账 |

关键约束：目标用户中包含**律师、财务、医护**等隐私敏感人群，其素材（合同、病历、转账记录）**在合规上不允许上云**。这直接决定了技术路线——必须端侧闭环。

---

## 二、用户痛点

| 痛点 | 现状 | 本方案的回应 |
| :--- | :--- | :--- |
| **隐私焦虑** | 主流 OCR/语音 App 默认上传云端处理，敏感数据离开设备 | 全链路端侧推理，基础模式**零网络请求**，卸载即清除 |
| **信息沉没** | 截图/录音存了等于没存，不会主动转化为行动 | OCR/ASR → 结构化 → 一键写日历，**聊完即存，到点提醒** |
| **手动录入成本高** | 看一眼截图再手敲进日历/备忘录，3~5 步操作 | AI 自动提取，核心闭环 **≤3 步** |
| **识别不可控** | 纯自动化提取出错无从修正 | 人机协同：低置信度字段红框高亮、模糊时间强制补全 |
| **离线能力缺失** | 多数 AI 工具断网即不可用 | 飞行模式下全流程可用，无网络依赖 |

## 三、技术方案

### 3.1 模型选型

三个模型各司其职，按"轻量专用"原则选型，控制总包体与内存占用：

| 环节 | 模型 | 选型理由 |
| :--- | :--- | :--- |
| **图像文字识别** | **PP-OCRv6_small**（MNN 版，det 4.7MB + rec 10MB） | 检测+识别双模型总计 ~15MB，中文识别精度高、小模型适配中低端机；纯卷积结构，CPU 推理延迟可控 |
| **音频理解/转写** | **Gemma 3n E2B**（MNN 多模态版） | 原生支持 `<audio>` 输入，端侧可跑，转写 + 语义理解一体，避免独立 ASR + NLU 两段式误差累积 |
| **结构化推理** | **Qwen3-2B**（MNN HQQ 量化版） | 2B 量级在端侧延迟/精度平衡点最佳；支持 `/no_think` 关闭推理直出 JSON；指令遵循能力足以稳定输出固定 schema |

> **选型权衡说明（诚实评估）**：PRD 早期文稿曾写作 "PP-OCRv5 / Qwen3.5-2B"，工程实际落地采用社区已有的 **PP-OCRv6_small** 与 **Qwen3-2B** MNN 量化权重——这是可获得、可验证的真实权重。多模态音频模型对内存敏感，4GB RAM 机型上需配合 `MNN_LOW_MEMORY` 与 mmap 加载，属待真机验证项。

### 3.2 推理框架：为什么是 MNN

MNN 是阿里开源的**端侧推理引擎**（非训练框架），三个模型可统一运行在同一引擎上，避免引入多套 runtime：

- **统一栈**：OCR（CV/卷积）与 LLM（Transformer）共用一套 `libMNN.so`，OCR 走 Module API（`Express::Module`），LLM 走 MNN-LLM 引擎（`LlmSession`/JNI），共享 `RuntimeManager`。
- **异构后端调度**：CPU（ARM82 / fp16）为主，可选 OpenCL GPU 卸载；图优化 + NC4HW4 内存布局（按 4 通道打包做 SIMD）。
- **SME2 加速路径**：在支持 SME2 的新一代 ARM CPU（如骁龙 8 Elite）上，编译期开启 `-DMNN_SME2=ON -DMNN_KLEIDIAI=ON` 后，GEMM 走 SME2 优化路径，矩阵乘显著提速——这是 LLM 解码阶段的核心算子。
- **HQQ 量化**：权重高精度量化，兼顾速度与精度，配合 `MNN_CPU_WEIGHT_DEQUANT_GEMM` 在线反量化。

### 3.3 端侧适配思路

整体处理链路（断网可跑）：

```
截图/拍照 ─→ PP-OCRv6 识别 ─┐
                              ├─→ Qwen3-2B 结构化 ─→ JSON ─→ 确认页 ─→ 日历/备忘录/归档
录音      ─→ Gemma 3n 转写 ─┘
```

四项关键端侧适配决策：

1. **图像解码绕开 MNN 解码器**：Android 端用系统 `BitmapFactory` 解码（对 JPEG/HEIC/PNG 及 Exif 旋转最可靠），再把裸 RGB buffer 交给 native，避免 MNN 图像解码器对 HEIC 等格式的兼容坑（见 `OcrEngine.kt` / `ocr_engine.h::runRgb`）。

2. **OCR 走原生 C++ 而非 Java 层**：检测（DBNet：概率图二值化 `kDetThresh=0.3` → 连通域 → 最小外接四边形 → unclip）+ 识别（透视裁剪矫正 → CTC greedy 解码）全部在 `ocr/ocr_engine.cpp` 用 MNN-CV 算子实现，长边限制 960、识别高度 48，按"上→下、左→右"排序还原阅读顺序。

3. **LLM 复用现成 MNN-LLM 通道**：结构化与音频转写不另起炉灶，直接复用工程已有的 `ChatService.createLlmSession` / `LlmSession` JNI——结构化模型 `updateThinking(false)` + `maxNewTokens=256` 压低延迟，音频模型 `supportOmni=true` 走多模态 `<audio>path</audio>` 输入。

4. **Prompt 工程保证结构化可解析**：单一 JSON 输出约束 + `/no_think` 硬保险 + 鲁棒 JSON 抽取（容忍 markdown 包裹/前后缀），解析失败时降级为"原文前 15 字 + 标记低置信度"，绝不崩溃。

## 四、创新点

1. **三模型异构流水线，单引擎统一调度**
   CV 模型（PP-OCRv6）与两个 LLM（Gemma 多模态、Qwen 文本）在同一 `libMNN.so` 内协同，这是端侧少见的"OCR + ASR + 结构化"三段式纯本地链路，工程上证明了 MNN 同时承载 CV 与 LLM 负载的可行性。

2. **"识别即结构化"而非"识别即展示"**
   绝大多数端侧 OCR 止步于出文字。本方案把 OCR/ASR 的输出直接喂给 LLM 做**场景判别 + 字段抽取**（chat/meeting/card/invoice/note 六场景自适应），让碎片信息一步到位变成可执行的结构化台账。

3. **隐私可证明，而非隐私承诺**
   "零联网"不是口号——基础模式下无任何网络代码路径被触发，云端增强为独立可选模块且默认关闭、强制脱敏、原图/原录音禁止上传，密钥走 Android Keystore。隐私是架构级保证。

4. **人机协同的"可信降级"设计**
   低置信度字段红框高亮、"下周/月底"等模糊时间弹出日期选择器强制补全、JSON 解析失败优雅降级——把不可避免的端侧小模型误差，转化为用户一眼可校的协同点，而非静默出错。

5. **SME2 就绪的前瞻性**
   编译链路预置 SME2 开关，在新一代 ARM CPU 上 LLM 解码可直接吃到指令集红利，方案随硬件演进自动受益。

## 五、预期效果

### 5.1 性能目标

| 指标 | 目标值 | 备注 |
| :--- | :--- | :--- |
| 单图 OCR + 结构化 | ≤5s | 旗舰机型；中低端机型预计更高 |
| 音频转写（1 分钟） | ≤5s | 多模态模型，内存敏感 |
| 批量 20 张处理 | ≤30s | |
| 千条记录搜索 | ≤0.5s | Room/SQLite 本地索引 |
| 应用冷启动 | ≤2s | 模型按需懒加载 |

> 上述为设计目标，需在 4GB+ RAM、Android 10+ 真机基准（`llm_bench`）实测确认；多模态音频在低内存机型上的延迟为主要风险项。

### 5.2 用户价值

- **隐私敏感行业可用**：律师/财务/医护可放心处理合同、账单、病历类素材。
- **闭环效率**：从"看到信息"到"已进日历并设好提醒"，3 步内完成。
- **离线可靠**：飞行模式、地下室、跨境无网环境照常工作。
